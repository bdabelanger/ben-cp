<h2>
 Problem Definition
</h2>
<h2 local-id="0cffe71b5fd2">
 Context
</h2>
<p local-id="bf7ead5ae7cb">
 The existing note creation experience in Casebook is a modal dialog that opens additional overlapping dialogs as users add People, Services, and Attachments to a note. This causes performance issues and a disjointed user experience - particularly for high-volume users who create service notes with many recipients, services, and attachments in a single session.
</p>
<p local-id="e3389c78e652">
 The new note dialog is a redesigned, full-featured data entry experience built from scratch that consolidates all note data into a single non-overlapping dialog. It replaces the legacy "Add note" modal across Engage, Intake, Track, and the Dashboard.
</p>

<h3 local-id="9bdd9b7383b5">
 Current note creation experience
</h3>
<p local-id="ee5a1c9c8f0f">
 Today, note creation is handled by a legacy modal dialog with the following limitations:
</p>
<ul local-id="3b3067559f91">
 <li local-id="a758e57e671d">
  <p local-id="38d14175ebe6">
   Adding People, Services, or Attachments opens a separate overlay dialog on top of the note modal, creating up to two or three nested modals simultaneously
  </p>
 </li>
 <li local-id="18fb804ee915">
  <p local-id="99e725c8830c">
   Multiple stacked modals degrade performance and are disorienting for users
  </p>
 </li>
 <li local-id="92a0d5dfa3ed">
  <p local-id="83ff54ebae21">
   The dialog is not built to accommodate growing feature requirements (Dynamic Pages, service rosters, access controls, labels/templates)
  </p>
 </li>
 <li local-id="11e77494d7f8">
  <p local-id="4f68b46016ef">
   There is no auto-save - users who navigate away lose their work silently (
   <a href="https://casecommons.atlassian.net/browse/CBP-2912">
    CBP-2912
   </a>
   )
  </p>
 </li>
</ul>
<h3 local-id="a0fe3d5cab90">
 Why rebuild from scratch
</h3>
<p local-id="ef9d3266661e">
 The new dialog is being built from scratch rather than patching the legacy modal for four key reasons:
</p>
<ol local-id="92094b3e1881" start="1">
 <li local-id="675b092cd29c">
  <p local-id="b9f89b45d872">
   <strong>
    Foundation for future features
   </strong>
   - Service rosters, service groups, dynamic pages for notes, labels/templates, and Access controls all require the new dialog architecture to be feasible
  </p>
 </li>
 <li local-id="8d7be9d4dbb6">
  <p local-id="9391b445f31f">
   <strong>
    Different data types in individual components
   </strong>
   - People, Services, Attachments, and Access each require specialized input components (datagrid rows, file upload, autocomplete) that do not compose well within the legacy modal architecture
  </p>
 </li>
 <li local-id="e1d1879db6ac">
  <p local-id="ad004f0fb616">
   <strong>
    Minimize overlapping dialogs
   </strong>
   - All sections are rendered inline within a single dialog, eliminating nested modals and their associated performance and usability issues
  </p>
 </li>
 <li local-id="747aaef8c016">
  <p local-id="ce7f332cfa1a">
   <strong>
    Clean slate for testing
   </strong>
   - The rebuild enables extensive internal testing without risk of breaking production functionality
  </p>
 </li>
</ol>
<h2 local-id="0ca32df20189">
 Problems &amp; Impact
</h2>
<h3 local-id="d7b0afd2f76e">
 Limited foundation for future capabilities
</h3>
<p local-id="1ceb24c917af">
 The legacy dialog cannot cleanly support planned capabilities including: service rosters (bulk-add recipients to a service), access controls per note, label-based note templates, and dynamic page sections. The new dialog is architected to accommodate these.Nested dialogs degrade performance and UX
</p>
<p local-id="ed75a4ada03b">
 When adding People or Services to a note, the current implementation opens a second modal on top of the note creation modal. Opening multiple modals simultaneously has been associated with rendering delays and increased error rates in high-volume tenants.
</p>
<h2>
 Solution Concept
</h2>
<h2 local-id="70c5b8bf47d2">
 Overview
</h2>
<p local-id="859e12a7ccc1">
 The new note dialog is a rebuilt, full-screen-style dialog that renders all sections inline - Note details, People, Services, Attachments, and Access - without using child modals. It supports auto-save to prevent data loss and is feature-flagged for controlled rollout.
</p>
<h3 local-id="483ee2e2b425">
 Redesigned Note dialog
</h3>
<p local-id="e992e072aab8">
 The new dialog (
 <a href="https://www.figma.com/design/HcwaonV58UklaK8NIgXEGN/Notes-redesign?node-id=38-7615&amp;t=NbQaB30TOD13nx3X-1">
  Figma - Notes redesign
 </a>
 ) replaces the legacy modal. It is built behind separate feature flags per section so each section can be released and validated independently.
</p>
<p local-id="46b53f8e2144">
 <strong>
  Sections in the new dialog:
 </strong>
</p>
<ol local-id="e7a1ff436ece" start="1">
 <li local-id="0d3a71748497">
  <p local-id="8d0e31ddf09e">
   <strong>
    Note details
   </strong>
   - Note type, Subject, Narrative (rich text), Start/end dates, Contact method / Delivery method (contextual by note type)
  </p>
 </li>
 <li local-id="1f0922a6fd1c">
  <p local-id="3173f871b346">
   <strong>
    People
   </strong>
   - Inline datagrid for adding and removing people involved in the note (
   <a href="https://casecommons.atlassian.net/browse/CBP-2921">
    CBP-2921
   </a>
   )
  </p>
 </li>
 <li local-id="2d1a1bbaed6f">
  <p local-id="6ae70e066fd1">
   <strong>
    Services
   </strong>
   - Inline datagrid for associating services to a service note, including per-person attendance (
   <a href="https://casecommons.atlassian.net/browse/CBP-2924">
    CBP-2924
   </a>
   )
  </p>
 </li>
 <li local-id="598222f607c4">
  <p local-id="dba1270c011e">
   <strong>
    Attachments
   </strong>
   - Inline file upload area replacing the legacy "Note attachments" overlay dialog (
   <a href="https://casecommons.atlassian.net/browse/CBP-2926">
    CBP-2926
   </a>
   )
  </p>
 </li>
 <li local-id="cfe63c2daa87">
  <p local-id="5af4e156ad9c">
   <strong>
    Access
   </strong>
   - Controls note visibility and access permissions (
   <a href="https://casecommons.atlassian.net/browse/CBP-2923">
    CBP-2923
   </a>
   )
  </p>
 </li>
</ol>
<p local-id="0bb060a2ba13">
 <strong>
  Auto-save
 </strong>
 (
 <a href="https://casecommons.atlassian.net/browse/CBP-2925">
  CBP-2925
 </a>
 ) - The dialog auto-saves note content at regular intervals and on section changes, ensuring that partial notes are not lost if the dialog is closed unexpectedly.
</p>
<h3 local-id="47e0fc539ddd">
 Consistent date fields
</h3>
<p local-id="95a9a03d2c47">
 Note start/end dates and Service start/end dates are now unified in the new dialog, with a migration to populate date fields from legacy service interaction data (
 <a href="https://casecommons.atlassian.net/browse/CBP-3105">
  CBP-3105
 </a>
 ). Date fields are renamed for clarity: "Note start date" / "Note end date" in the Note section, "Service start date" / "Service end date" in the Services section (
 <a href="https://casecommons.atlassian.net/browse/CBP-3107">
  CBP-3107
 </a>
 ).
</p>
<h3 local-id="5b94f1810bd7">
 Service groups and rosters (future)
</h3>
<p local-id="2942c86c3a19">
 Future iterations of the Services section will support service rosters (bulk-adding multiple recipients to a single service enrollment) and service group selection (
 <a href="https://casecommons.atlassian.net/browse/CBP-3083">
  CBP-3083
 </a>
 ,
 <a href="https://casecommons.atlassian.net/browse/CBP-3113">
  CBP-3113
 </a>
 ).
</p>
